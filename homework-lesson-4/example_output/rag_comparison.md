# Порівняння підходів до побудови RAG: Naive, Sentence-Window та Parent-Child

У цьому звіті розглядаються три основні архітектурні підходи до побудови систем Retrieval-Augmented Generation (RAG), які відрізняються за складністю, точністю пошуку та контекстуальною повнотою.

---

## 1. Naive RAG (Базовий підхід)

Це класична архітектура RAG, яка є відправною точкою для більшості розробників.

*   **Принцип роботи:** Документи розбиваються на фіксовані за розміром чанки (chunks), які індексуються у векторній базі даних. Під час запиту система шукає найбільш схожі за семантикою чанки та передає їх у LLM.
*   **Переваги:**
    *   Простота реалізації.
    *   Мінімальні вимоги до інфраструктури.
*   **Недоліки:**
    *   **Втрата контексту:** Чанки можуть бути розірвані посеред речення або думки.
    *   **Низька точність:** Якщо запит вимагає ширшого контексту, ніж один чанк, модель не отримає достатньо інформації.

## 2. Sentence-Window Retrieval

Цей підхід спрямований на вирішення проблеми "вузького контексту" при збереженні точності пошуку.

*   **Принцип роботи:** Документи розбиваються на окремі речення. Кожне речення індексується окремо. Однак, коли система знаходить релевантне речення, вона витягує не тільки його, а й "вікно" (кількість речень до та після нього).
*   **Переваги:**
    *   **Висока точність пошуку:** Векторний пошук працює на рівні речень, що дозволяє дуже точно знаходити відповідні фрагменти.
    *   **Багатий контекст:** LLM отримує не просто "шматок тексту", а логічно завершений контекст навколо знайденої інформації.
*   **Недоліки:**
    *   Потребує додаткової логіки для керування розміром вікна.
    *   Може бути надлишковим, якщо вікно занадто велике.

## 3. Parent-Child Retrieval (Small-to-Big)

Цей підхід використовує ієрархічну структуру даних для оптимізації пошуку.

*   **Принцип роботи:** Документ розбивається на великі "батьківські" чанки (parent chunks), які потім діляться на менші "дочірні" чанки (child chunks). У векторну базу індексуються лише дочірні чанки, але вони містять посилання на свій батьківський чанк. При пошуку система знаходить дочірній чанк, але передає в LLM весь батьківський чанк.
*   **Переваги:**
    *   **Баланс:** Дозволяє використовувати переваги точного пошуку (малі чанки) та повноти контексту (великі чанки).
    *   **Масштабованість:** Добре працює з великими документами, де важливо зберегти цілісність розділів.
*   **Недоліки:**
    *   Складніша архітектура індексації.
    *   Вищі витрати токенів (оскільки в LLM передається великий батьківський чанк).

| Характеристика | Naive RAG | Sentence-Window | Parent-Child |
| :--- | :--- | :--- | :--- |
| **Складність** | Низька | Середня | Висока |

## Висновок

*   **Naive RAG** підходить для простих прототипів або дуже коротких документів.
*   **Sentence-Window** ідеальний, коли важливо знайти конкретну відповідь у великому масиві тексту, не втрачаючи при цьому зв'язку з навколишніми реченнями.
*   **Parent-Child** найкраще підходить для складних баз знань, де відповідь на запитання часто вимагає розуміння цілого розділу або документа.
